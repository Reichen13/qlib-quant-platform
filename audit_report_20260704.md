# 量化系统审查报告

**审查日期:** 2026-07-04
**审查对象:** Qlib Quant Platform(D:\qlib),作为"A股个人波段交易者(10~50万,佣金万2.5+最低5元,无两融,行为回撤承受约20%,持仓几天~几周)的实战工具"
**审查方法:** 先写基准尺(见 `audit_baseline_20260704.md`),后读代码;所有结论附 文件:行号 或本机数据样本证据;高杀伤结论已二次交叉核验。

## 0. 一句话结论

**这套系统当前不合格作为实战工具:本机 Qlib 数据在拼接边界和除权事件处已系统性损坏(98.2% 的股票存在 >11% 的假跳空,茅台等 61 只核心票数据空心),回测显示的是零摩擦毛收益且你设置的止损/调仓参数根本没接进引擎——在数据管道修复之前,平台产出的任何回测收益率和技术信号都不应作为下单依据。** 作为研究原型,它的工程骨架(任务持久化、无模拟数据兜底、IC 前向对齐)是真实合格的,值得修而不是弃。

## 1. 关键发现 · 按 P&L 杀伤力排序

### 🔴 #1 数据主干缺陷链:前复权增量只追加不重写 + 复权因子占位 1.0 + 多源随机缝合 + 日历错位 —— 全市场级 K 线污染

**杀伤力:高(所有下游的地基)**

**证据(代码):**
- `update_cn_data.py:457-463`:增量只追加 `cal_idx > end_idx` 的新行,**从不重写旧段**。前复权数据每次分红除权后全历史漂移,只追加意味着新旧段复权基准不同。
- `update_cn_data.py:147/186/249/307`:四个数据源写入的复权因子一律 `df["factor"] = 1.0`(占位),真实除权因子不可审计、不可恢复原始价;`:391` 修复逻辑明确永不覆盖 factor。
- `update_cn_data.py:315-326`:同一只票按 腾讯→yfinance→baostock→东财 逐次失败切换,**数据段来源取决于当次网络状况**,四个源的"前复权"算法并不等价,且无来源记录。
- `update_cn_data.py:56-68`:`extend_calendar` 会对 day.txt 排序去重后整体重写,而 bin 只存日历索引——日历中段一旦变化,所有旧 bin 日期映射整体错位。

**证据(本机数据样本):**
- 3606 只跨 2020-09-25/28 拼接边界的股票中,**3541 只(98.2%)边界单日"涨跌"超过 11%**;sz000001:2020-09-25 close=3.17 → 2020-09-28 close=12.63(假涨 +298%);最极端 sz300866:0.94 → 95.05。
- sz000001 增量段内部同样漂移:bin 存值与今日腾讯前复权同日偏差 -13%~-17% 且比率逐段变化(多次拉取不同基准缝合的直接证据)。
- **61 只 bin 长度>3000 但有效值<200 的"空心票"**,含 sh600519/sh600036/sh601318/sh510300。sh600519 仅有两段数据:1999-11-10~12-09 存着 1459 元级价格(茅台 2001 年才上市——2026 年行情被错位日历索引写进了 1999 年),外加 2026-04-29 之后 44 天。
- 健康检查只看 bin 末日索引(`backend/api/data.py:69-127`),空心茅台末日=2026-07-03 → 判"新鲜正常";跳变探测阈值 35%(`data.py:176`),分红级 1%~15% 假跳空全部漏检。

**用户视角影响:** 你的波段信号(均线、MACD、突破、ATR 止损)全部建立在含假跳空、可能错位、复权基准逐段漂移的 K 线上。回测赚的钱是数据裂缝里长出来的,实盘对不上;更危险的是页面显示"数据正常",你会信任它。

**修复:** 见 90 天行动第 1 件事。核心是改存"不复权原始价 + 真实复权因子",增量改为除权事件后全历史重写,日历禁止中段插入,本机 61 只空心票立即全量重建。

### 🔴 #2 回测报表是零摩擦毛收益:配置的交易成本一分钱没扣,而策略实际每天换 20% 仓

**杀伤力:高(直接决定"回测赚钱→实盘亏钱")**

**证据:**
- pyqlib `qlib/backtest/report.py:32` 明文:`return: ... **without transaction fee**`;`backend/api/backtest.py:480` `r = report["return"]` 之后所有指标(总收益/年化/夏普/回撤/净值曲线)全部用毛收益,全文件无一处减 `report["cost"]`(已二次 grep 核验)。`run_alpha158.py:170` 还把它标注为"Sharpe(含成本)",标签失实。
- `backend/api/backtest.py:455-456`:`n_drop = max(1, topk // 5)`,TopkDropoutStrategy 每个交易日执行 → 默认 hold_num=30 时**每日单边换手 ≈20%**。你在界面填的 `turnover`(调仓周期,默认 5 天)只用于一段文本估算(`:540`),**从未传入策略**。
- 成本参数本身也低估:`sell_cost` 默认 0.0003(`backend/models/schemas.py:162-163`),真实卖出 ≈0.076%(佣金万2.5+印花税0.05%+过户费),低估 60%。

**计算复现(隐藏摩擦量级):** 年摩擦 ≈ 252 × 日单边换手 × 往返费率。
- 50 万资金、真实费率往返 ≈0.102%:252 × 20% × 0.102% ≈ **5.1%/年**;
- 10 万资金 × 30 只 = 每笔 ≈3167 元,最低佣金 5 元把单边费率抬到 0.158%,往返 ≈0.368%:252 × 20% × 0.368% ≈ **18.5%/年**。
- A 股多因子 TopK 年化超额通常在个位数到 15%——**摩擦足以吞掉全部超额,而回测界面显示的是 0 摩擦毛收益**。

**用户视角影响:** 你看到"年化 15%、夏普 1.5"决定投钱,实盘拿到的可能是负数,而且你不会知道差在哪。

**修复:** `backtest.py:480` 起改用 `r_net = report["return"] - report["cost"]`,毛/净两条曲线同屏展示;`sell_cost` 默认改 0.0008;调仓周期真正接入策略(见行动 2)。

### 🔴 #3 智能股票池的"ICIR 加权"从未生效,实际是 158 个因子等权平均 ≈ 噪声选股

**杀伤力:高(这是直接产出选股名单的链路)**

**证据:**
- `backend/core/stock_pool.py:271` 读取 `~/.qlib/cache/factor_icir.parquet` —— **全仓 grep 仅此一处出现该文件名,没有任何代码写入它**(已二次核验),因此 ICIR 权重恒为空。
- 恒走 `stock_pool.py:241-246` 降级分支:对全部 158 个**原始未标准化**因子取等权算术平均。动量类与反转类因子符号相反互相抵消,不同量纲因子相加被大数值因子主导,且未按 IC 方向调向——分数接近噪声。
- 界面/文档宣称"ICIR 加权 + 行业中性化",名不副实;降级是静默的,无任何 warning。

**用户视角影响:** 你以为在用"智能因子加权选股",实际拿到的候选 30 只近似随机排名。10~50 万按这个名单建仓 = 无 alpha 裸暴露。

**修复:** 因子分析任务完成时把 ICIR 写入该 parquet(analyze 已经算出 `icir`);缓存缺失时禁止静默等权,响应里加显式 warning;等权分支至少先截面标准化+按 IC 方向调向。

## 2. 四层诊断

### 2.1 数据层

- **理想形态:** 每根 K 线可回答复权方式与来源;样本带生命周期(上市/ST/停牌/退市);更新幂等;缺失显式标记;历史可复现。
- **现状评估:不合格。**
- **缺陷清单(按杀伤力):** #1 主干缺陷链(拼接假跳空/factor 占位/多源缝合/日历错位/空心 bin);#8 幸存者偏差双向;停牌日以"0 量+前值+0% 涨跌"假行情行存在(本机抽样 92/200 只)且无 tradestatus 标记;ST 历史无记录(仅按当前名称过滤,`stock_pool.py:146`);次新/市值/资不抵债过滤是空壳参数(`stock_pool.py:64-66` 有定义、`:136-152` 无实现);财务数据按当前自然季度查报告期(7 月查 Q3 必空,`data_provider.py:244-247`),pubDate 返回但未用;健康检查"新鲜"≠可用(#1 内)。
- **正面:** 追加层面幂等;中间缺口填 NaN 不填 0;无模拟数据兜底。
- **修复优先级:** 先修 #1(其他一切的地基)→ 健康检查加有效值密度指标 → 停牌/退市标记 → 空壳过滤参数要么实现要么标 unavailable。

### 2.2 因子层

- **理想形态:** 因子有身份证;半衰期匹配 5~20 日持仓;截面可比(行业+市值中性);无未来函数;知道共线簇。
- **现状评估:部分合格。** 骨架(IC 对齐、聚类去重、衰减分析)真实且正确,但默认口径让结果对你不可用。
- **缺陷清单:** #3 股票池 ICIR 失效(最重);#9 默认口径(CSI300/前20因子/无中性化);#10 retail_factors 拼凑因子;技术指标缺价填 0(`quote.py:55-60`)+ 按行序取窗口忽略停牌间隔(`quote.py:115-145`);Alpha158 默认 Fillna(0) 压缩排序区分度;完全没有基本面因子(等价于基本面输入为空)。
- **修复优先级:** #3 → #9 的 top_k 计算截断(改为全量计算、仅展示截断)→ 默认开中性化或显著标注 → retail_factors 改名或下线。

### 2.3 回测层

- **理想形态:** T+1/涨跌停/停牌/一字板保守撮合;成本与真实账户一致;样本外+参数稳定性;回测≠策略。
- **现状评估:不合格。** 值得肯定:主回测路径的 train/test 泄漏防护是真实做对了的(`build_leak_safe_segments` 有 label embargo,`fit_start/end_time` 严格等于训练段,`backtest.py:117-159, 360-361`),T+1 结构成立,停牌 NaN 不可成交。但:
- **缺陷清单:** #2 毛收益零成本;#5 参数无效+涨跌停一刀切+100 万硬编码+零滑点无限量;无样本外/walk-forward/±20% 参数扰动(单一切分,valid 仅早停);归因只有行业 Brinson 且基准口径不一(等权近似 vs 市值加权,`backtest.py:1008` 自认);约束统计失败静默归零(`backtest.py:232-234, 256-259`——停牌/涨跌停命中数 except 后报 0);dl_models 训练段与测试段重叠且无 embargo(`core/dl_models.py:318-321`,虽自称桩实现);#10/#11 的 pair/turtle/mean_reversion 四个信号模块零回测验证;回测全量 pred 不落盘(只存最后一天 Top-K),换仓决策不可复核。
- **修复优先级:** #2 → #5 → 约束统计失败改报"未知"而非 0 → 简版 walk-forward。

### 2.4 工程层

- **现状评估:部分合格。** 这是四层里做得最好的一层:任务 SQLite 持久化+重启标记中断原因、长假处理锚定真实交易日历、外部源失败逐条 warning 透出前端、密钥未入仓、数据更新接口鉴权,均属实。

## 5. 不建议做的事(洁癖项,明确不必碰)

1. **不要重写回测框架。** Qlib 的 Exchange/TopkDropout 本身支持你需要的一切(分板涨跌停表达式、volume_threshold、成本入账)——问题全在调用参数,改 kwargs 即可,自研框架是最大的时间黑洞。
2. **不要上分布式/K8s/feature store/实时行情流。** 你是单用户盘后系统,SQLite + 单容器完全够用,现有任务持久化已经合格。
3. **不要新增因子和新增策略源。** 16 个信号源已经太多,90 天内正确方向是做减法(聚焦层)而不是加法。基本面因子可以等数据层修完再考虑,且必须用公告日对齐。
4. **暂停对 LLM 功能(智能体辩论/AI 策略/新闻情绪)的投入。** 它们不在任何一条 P&L 关键链路上,且 LLM 评分混入筛选还破坏可复现性(`screening.py:632-635`)。修完前五件事之前,每一小时花在这里都是从数据修复那里偷来的。
5. **不要清理根目录几十个死脚本**(纯洁癖)——但有两个例外要处理:`data_collector.py` 和 `app.py`/`start*.sh` 是误导性双入口,各加一行"已弃用,请用 update_cn_data.py / docker compose"的头部声明即可,不必删除。
6. **dl_models 不必完善。** 它自称桩实现且训练段泄漏,在因子/回测层修好之前,深度模型是负优先级;界面上标注"实验性,勿用于实盘"即可。
7. **不需要机构级归因体系。** 现有 Brinson 行业三分解够用,只需把归因基准与收益基准口径统一的那行注释变成真实修复,风格/择时归因不做。
8. **写接口鉴权不急。** 127.0.0.1 绑定 + 单用户场景下杀伤力低,排在所有 P&L 相关修复之后。
