## 一、致命缺陷：4 个模块使用完全虚假数据

这是本轮审查最重要的发现。以下模块的数据**不是从真实行情计算出来的**，而是硬编码的静态数字：

### 🔴🔴 1. ETF 轮动模块（`backend/api/etf.py`）

**问题严重度：致命**

```python
# 第45-66行：静态硬编码涨跌幅
ETF_BASE_CHANGE = {
    "SH510300": 0.4,    # 这些数字永远不会变
    "SZ159995": 1.8,
    ...
}

# 第69-80行：信号逻辑只看这个静态字典
def get_etf_signal(code, days=20):
    change = ETF_BASE_CHANGE.get(code, 0)
    if change >= 1.0: return "buy"
    ...

# 第104行：价格用 index 算出来的
price=round(1 + index * 0.03, 3)  # 第0个ETF价格=1, 第1个=1.03...
```

**后果：** 用户看到的 ETF "信号"永远不变——无论真实市场涨跌。前端还做了 `refetchInterval: 60000`（每60秒刷新），每次拉到的都是同样的假数据。

**修复方向：** 从 yfinance/Qlib 获取真实 ETF 日线，计算真实动量信号。

### 🔴🔴 2. 行业板块模块（`backend/api/sectors.py`）

**问题严重度：致命**

```python
# 第18-39行：20个板块的涨跌幅全部硬编码
SECTOR_BASE_CHANGE = {
    "半导体": 2.8,
    "地产": -1.6,
    ...
}

# 第48行起：每只成分股的"涨跌幅"从 base 加随机噪声
change = info["base_change"] + random.uniform(-1.5, 1.5)
```

**后果：** 板块涨跌幅排名、热门板块、板块内股票表现——全部是假的。用户点击"半导体板块涨 2.8%"时，这个数字来自代码里的常量，不是市场数据。

**修复方向：** 用真实成分股收盘价计算板块加权涨跌幅。

### 🔴🔴 3. 配对交易模块（`backend/api/pair.py`）

**问题严重度：致命**

```python
# 第21行起：DEFAULT_PAIRS 全部硬编码
DEFAULT_PAIRS = [
    {"pair": "招商银行 / 平安银行", "correlation": 0.92, "pValue": 0.001, "zScore": 2.35, ...},
    {"pair": "贵州茅台 / 五粮液", "correlation": 0.88, ...},
    ...

```

**后果：** 相关性、p值、zScore 全是写死的。协整关系在真实市场中会随时间变化，但这里永远不会变。用户可能根据过期信号交易。

**修复方向：** 用 Qlib `D.features()` 获取真实价格，实时计算协整检验 + zScore。

## 二、严重架构缺陷（影响可靠性和可维护性）

### 🔴 5. 回测任务存储在内存中——进程重启全部丢失

**位置：** `backend/api/backtest.py` 第 24 行

```python
backtest_tasks: dict[str, dict] = {}
```

这是一个**模块级字典**。uvicorn 重启（部署、OOM、崩溃恢复）时所有进行中和已完成的任务全部丢失。

**后果：**
- 用户提交回测 → 等待 2 分钟 → 服务重启 → 任务丢失 → 无任何通知
- 无法查看历史回测结果
- 多进程下不同 worker 看到不同的任务列表

**修复方向：** 至少用 SQLite 持久化（项目已有轻量依赖），理想用 Redis（Docker 加一个容器即可）。

---

### 🔴 6. 无任何用户认证/隔离

**位置：** `backend/main.py` 第 146-152 行

```python
app.add_middleware(CORSMiddleware, allow_origins=["*"], ...)
```

- 无登录、无 token、无 session
- 任何能访问 `http://server:8000` 的人都能触发回测、查看所有数据
- 多用户共享一个任务队列（回测任务没有 user_id 字段）

**后果：** 部署到公网 = 任何人随意使用你的计算资源 + 看到你的策略参数。

**修复方向：** 至少加一个简单的 API Key 验证（环境变量 `API_KEY`），生产环境需要 OAuth2/JWT。

---

### 🟡 7. 前端开发服务器用于生产

**位置：** `start_new.sh` 第 47 行

```bash
npm run dev -- --host 0.0.0.0 --port 5173 &
```

`vite dev` 不是为生产设计的——无压缩、无缓存、HMR 开销、source map 泄露。

**修复方向：** `start_new.sh` 应改用 `npm run build && npm run preview`，或直接由 nginx 提供静态文件。

---

### 🟡 8. 股票名称映射是静态字典——会过时

**位置：** `stock_names.py` 约 230 行硬编码

CSI300 每半年调样（6月和12月），新进入的股票在字典中找不到，只能回退显示代码。

**修复方向：** 用 Baostock `query_stock_basic()` 动态获取，缓存到内存/磁盘。

## 三、投资逻辑层面的重大缺陷

### 🔴 9. 回测买/卖推荐的"理由"是模板文本——不是真实分析

**位置：** `backend/api/backtest.py` `_generate_buy_reason()` 函数

回测结果中的 "推荐买入/卖出" 及理由是从因子名做字符串匹配生成的——"因子 score 高"、"因子 rank 领先"——而非基于归因分析或基本面数据。

**后果：** 即使 Brinson 归因显示超额收益全部来自行业配置（选股贡献为负），推荐理由仍然会写"选股能力突出"。

**修复方向：** 推荐理由应根据 Brinson 归因结果动态生成——"本期超额收益主要来自超配银行板块（配置贡献 +2.1%），选股方面白酒板块内选股贡献 +0.8%"。

---

### 🟡 10. ETF 轮动信号质量薄弱（即使真实数据后）

**来源：** 第二轮遗留 #11

当前信号基于简单涨跌幅阈值：
```python
if change >= 1.0: return "buy"
if change <= -0.5: return "sell"
return "hold"
```

缺少：
- 风险调整动量（动量/波动率）——高波动 ETF 天然容易触发 buy
- 趋势过滤（60日均线以下不做多）——熊市反弹假信号
- 相关性过滤——同时持有 5GETF + 通信ETF + 计算机ETF 高度重叠

**修复方向：** 这是第二轮 P3 遗留项，但对 ETF 模块的真实化是前提。

---

### 🟡 11. 相关性矩阵只为全样本——危机相依不可见

**来源：** 第二轮遗留 #8

当前 `_compute_correlation_matrix()` 使用**全样本 Pearson 相关**。实证显示 A 股下跌市的相关性比上涨市高 30-60%（" diversification breakdown "）。

**后果：** 用户看到的 "平均相关性 0.45" 在熊市可能是 0.75——分散化在需要它的时候失效了。

**修复方向：** 分上行市（月收益 > 0）和下行市（月收益 < 0）展示两套相关性矩阵。

---

### 🟡 12. 有效前沿只有一条曲线——无可行前沿对比

**来源：** 第二轮遗留 #13

当前有效前沿是在 `max_weight=30%` 约束下计算的一条曲线。但缺少：
- **理想前沿**（无约束 MV 边界）——展示理论最优
- **可行前沿**（加入流动性/行业/换手约束后）——展示实际操作空间

两条曲线之间的区域 = 约束成本，这是组合经理最关心的信息。

**修复方向：** 在同一图上画两条前沿曲线，标注约束成本。

---

## 四、数据质量与时间敏感性

### 🟡 13. App-Store 默认日期已过时

**位置：** `frontend/src/stores/app-store.ts`

```typescript
const DEFAULT_BACKTEST_PARAMS = {
  trainStart: "2023-01-01",   // 固定的
  trainEnd: "2024-06-30",
  testStart: "2024-07-01",
  testEnd: "2024-12-31",      // 已过去 5 个月
}

const DEFAULT_FACTOR_PARAMS = {
  startDate: "2026-01-01",    // 写了 2026 年
  endDate: "2026-04-30",
}
```

这些日期永远不会自动推进。用户每次打开页面看到的是 2024 年的回测区间。

**修复方向：** 默认日期应该动态计算——如 `trainEnd = 6个月前`, `testStart = 6个月前`, `testEnd = 昨天`。

---

### 🟡 14. 多数据源一致性无验证

**来源：** 第二轮遗留 #12

Qlib 数据、Baostock 日K、akshare 宏观——三个数据源各自独立，无交叉验证。同一标的 SH600519 在 Qlib 和 Baostock 的收盘价是否一致？无人检查。

**修复方向：** 在 `/api/data/health` 加入抽样交叉验证（差异 > 1% 告警）。

## 五、关键硬编码阈值审查

很多阈值有合理性（如 `limit_threshold=0.095` = A 股 10% 涨跌停板），但以下缺少注释或论证：

| 值 | 位置 | 问题 |
|----|------|------|
| `risk_free_rate=0.02` | risk.py | 当前中国 10 年期国债约 2.6-2.8%，2% 偏低 |
| `max_weight=0.3` | portfolio.py | 没有说明为什么是 30%（公募单票上限 10%） |
| `max_drawdown_day_threshold=-5%` | drawdown-chart.tsx | 为什么 -5% 算 "drawdown day"？无注释 |
| `min_cost=5` | backtest.py | A 股最低佣金 5 元，正确但无注释 |
| `account=1_000_000` | backtest.py | 为什么是 100 万？对散户偏高、对机构偏低 |
| `VIX>30` = panic | macro.py | 中国没有 VIX，用的是美股 VIX 做代理——效果存疑 |
| `PMI>50` = expansion | macro.py | PMI 季节性很强，需季节调整否则 3 月总是扩张 |

## 六、代码质量

### 🟢 15. 三个板块 API 功能重叠

- `api/hot.py` — "热门板块"
- `api/sectors.py` — "行业板块"  
- `api/industry.py` — "行业分析"

三者的 SECTOR_DEFINITIONS 不同（hot.py 有自己的硬编码列表，sectors.py 有另一份，industry.py 从 Qlib 获取）。同一用户访问三个端点，看到三个不同的板块列表。

### 🟢 16. 技术指标重复计算

后端 `quote.py` 和前端的 `quote/index.tsx` 都实现了 MA/RSI/MACD 计算逻辑。一端改了另一端不改就会不一致。

## 八、优先级建议（第三轮行动计划）

### 如果要修复，按这个顺序：

| 优先级 | 问题 | 理由 | 预估 |
|--------|------|------|------|
| **P0** | ETF/板块/配对/首页 假数据 → 真实 | 目前这 4 个模块的用户看到的是假信息，比"没有这个功能"更糟 | 2-3天 |
| **P0** | 回测任务持久化 | 重启丢任务 = 生产不可用 | 1-2天 |
| **P1** | 基本安全（API Key 或简单认证） | 公网部署直接裸奔 | 0.5天 |
| **P1** | 买/卖推荐理由基于归因动态生成 | 让 Brinson 归因的价值落地 | 1天 |
| **P1** | App-Store 日期动态化 | 用户每次打开看到过期日期 | 0.5天 |
| **P2** | 相关性矩阵分上涨/下跌市 | 风险管理实用性 | 1天 |
| **P2** | 有效前沿双曲线（理想 vs 可行） | 组合经理核心需求 | 1天 |
| **P2** | 生产部署优化（nginx静态 + 非dev模式） | 稳定性 | 0.5天 |
| **P3** | ETF信号质量/滚动协整/数据交叉验证/板块API合并 | 锦上添花 | 各1天 |

### 核心判断

**平台现状：** 因子研究链（Alpha158 → IC → 回测 → 归因）已达到**专业可用**水平，组合优化具备**机构级稳健性**（Ledoit-Wolf + JS + 换手约束）。

**最大短板：** 4 个假数据模块让平台在用户眼中从 "研究工具" 降级为 "技术演示"。如果用户先打开 ETF 轮动或首页——他们看到的是硬编码数字——就不会信任回测结果，无论回测做得再好。

**一句话：因子研究和回测已经可以做真实决策了，但需要先把假数据模块修好，别让它们破坏整体可信度。**
